<interfaces>
<!-- Admin API endpoints to consume -->

Channels:
  GET    /api/admin/channels           → [{id, name, description, thumbnail, category_ids, age_group_ids, ...}]
  POST   /api/admin/channels           ← {name, description, thumbnail, category_ids, age_group_ids} → 201
  GET    /api/admin/channels/{id}      → channel object
  PUT    /api/admin/channels/{id}      ← {name, description, thumbnail, category_ids, age_group_ids} → 200
  DELETE /api/admin/channels/{id}      → 204

Episodes:
  GET    /api/admin/episodes           → [{id, channel_id, title, description, order, status, ...}]
  GET    /api/admin/episodes?channel_id=xxx → filtered by channel
  POST   /api/admin/episodes           ← {channel_id, title, description, order, subtitle_url, source_url} → 201 or 202 (if source_url present)
  GET    /api/admin/episodes/{id}      → episode object
  PUT    /api/admin/episodes/{id}      ← {title, description, order, subtitle_url} → 200
  DELETE /api/admin/episodes/{id}      → 204

Jobs:
  GET    /api/admin/jobs               → [{id, episode_id, source_url, status, error, started_at, completed_at, ...}]
  GET    /api/admin/jobs?status=xxx    → filtered by status
  GET    /api/admin/jobs/{id}          → job object
  PATCH  /api/admin/jobs/{id}/retry    → 200 (resets to pending)

YouTube Metadata:
  GET    /api/admin/youtube-meta?url=xxx → {title, description, thumbnail, duration}

Categories (for channel form dropdowns):
  GET    /api/admin/categories         → [{id, name}]

Age Groups (for channel form dropdowns):
  GET    /api/admin/age-groups         → [{id, name, min_age, max_age}]

Status badge colors (from CONTEXT.md locked decision):
  pending: yellow, downloading: blue, transcoding: purple, ready: green, failed: red
</interfaces>

    **Jobs page (`jobs/page.tsx`):**
    - Client component ('use client') — needs useEffect for polling
    - Polls `GET /api/admin/jobs` every 5 seconds using `useEffect + setInterval` (per locked decision)
    - Cleanup interval on unmount: `return () => clearInterval(interval)`

    - **Table layout:**
      - Columns: Episode (title or ID), Source URL (truncated), Status (StatusBadge), Started At, Completed At, Duration (computed)
      - Status filter dropdown at top: All, Pending, Downloading, Transcoding, Ready, Failed
      - Filter passes `?status=xxx` query param to the API

    - **Failed job handling (per locked decision):**
      - Failed rows are visually distinct (red-tinted background or left border)
      - Each failed row has an expandable section (click to toggle)
      - Expanded section shows: full error message text
      - Expanded section has a "Retry" button
      - Retry calls server action that PATCHes `/jobs/{id}/retry`
      - After retry, the job status resets to "pending" — next poll shows updated status

    - **Polling behavior:**
      - Start polling immediately on mount
      - If all visible jobs are in terminal state (ready or failed) and no filter is active, optionally reduce polling frequency to 30s to save resources (Claude's discretion)
      - On retry click, immediately re-fetch (don't wait for next interval)

    **Server actions (`actions/jobs.ts`):**
    - `retryJob(jobId: string)` — PATCH to `/jobs/{id}/retry` with Bearer token
    - This needs to be callable from the client component. Since server actions must be in a 'use server' file, import and call from the client component via form action or direct invocation.

    **Important:**
    - All fetch calls use `/api/admin/jobs` (absolute path through nginx) for client-side polling
    - Do NOT use apiServerFetch for polling — this is client-side
    - Use `apiFetch` (client-side helper) with credentials: 'include' so the admin_token cookie is sent
