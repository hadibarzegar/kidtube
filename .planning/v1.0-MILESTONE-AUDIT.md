## Integration Gaps

### Gap 1: Admin Client-Side Auth Wiring (Functional Bug)

**Root cause:** `admin-app/src/lib/api.ts` `apiFetch` sends `credentials: 'include'` (cookie) but does NOT set `Authorization: Bearer` header. `admin-api` only has `jwtauth.Verifier(auth.TokenAuth)` which reads `Authorization: Bearer` header — no `tokenFromCookie` extractor.

**Impact:**
- Admin jobs page polling loop returns 401 (VIDE-04 "live updates" broken)
- Any admin-app client-side fetch via `apiFetch` will fail authentication

**Contrast:** Site-api has `tokenFromSiteCookie` extractor registered — this pattern works for site-app. Admin-api lacks the equivalent.

**Fix:** Add `tokenFromAdminCookie` extractor to admin-api's `jwtauth.Verify` call, matching site-api's implementation.

### Gap 2: Site-App Route Guard Not Active (UX Bug)

**Root cause:** `site-app/src/proxy.ts` exports middleware but Next.js requires the file to be named `middleware.ts`. Build output confirms `middleware-manifest.json` has empty middleware object.

**Impact:**
- Guests navigating directly to `/subscriptions`, `/bookmarks`, `/account` see empty states instead of redirect to `/login`
- No data leak — backend returns 401, client components handle gracefully
- Interactive components (SubscribeButton, BookmarkButton) correctly redirect on 401

**Fix:** Rename `site-app/src/proxy.ts` to `site-app/src/middleware.ts`
